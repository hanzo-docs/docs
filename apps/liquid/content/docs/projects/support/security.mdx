# Security & Audits

Liquid sits inside the [Lux security stack](https://github.com/luxfi/security) —
a shared set of audit cadence, formal-verification harnesses, post-quantum
cryptography, MPC-backed key management, and disclosure paths used across
every Lux protocol. Liquid-specific surfaces are layered on top.

## Audit coverage

External audit reports live in [`luxfi/audits`](https://github.com/luxfi/audits)
(LaTeX, peer-reviewed). Per-component reports are tracked there with the
mainnet-deployment tag they cleared.

| Component | Audit | Source |
|-----------|-------|--------|
| Smart contracts | 2025-12-30 contracts audit | `luxfi/audits` |
| Cryptography | 2025-12-30 crypto audit | `luxfi/audits` |
| Architecture | 2025-12-30 architecture review | `luxfi/audits` |
| DEX VM | 2025-12-30 dexvm audit | `luxfi/audits` |

For Liquid-specific deltas (transmuter, position decay, strategy classifier),
re-audit scope is run before each major version bump.

## Internal security practices

- **Foundry unit + invariant tests** in `src/test/` and `src/test/Invariants/`
- **Foundry fuzzing** — 1000-run fuzz suite on every PR
- **Halmos** — symbolic execution over `check_*` properties (`make halmos`)
- **Slither + Aderyn + Semgrep** — static analysis on every PR (`make security`)
- **Multi-reviewer code review** — minimum two engineers + one security
- **Forked-mainnet integration tests** — strategy adapters run against mainnet state

```bash
make audit       # lint + tests + slither + semgrep + aderyn
make halmos      # symbolic property checking
make test-fuzz   # fuzz suite
```

## Upgradeability & governance

- **Core vault logic** (`Liquid.sol`, `LiquidPosition.sol`,
  `LiquidTransmuter.sol`) — immutable. No upgrade path.
- **Adjustable parameters** (caps, fees, classes, transmuter cadence) —
  Lux DAO governance.
- **Strategy adapters + curator/allocator/classifier/gauge** — admin role
  held by the Safe owned by `ModuleGovernorV1` (see
  [Governance](/docs/governance)).

Execution flow:

```
Proposal on lux.vote
  → DLUX-weighted vote (StrategyV1)
  → ModuleGovernorV1 queues + executes through Safe
  → Safe calls Liquid contract
```

## Post-quantum cryptography

Liquid inherits the Lux PQ stack from [`luxfi/crypto`](https://github.com/luxfi/crypto).
**NIST-PQC algorithms run in production today** with full crypto-agility — every
signing surface accepts both classical and PQ keys; verifiers dispatch on the
algorithm tag.

| Primitive | Spec | Use |
|-----------|------|-----|
| **ML-KEM-768** | FIPS 203 | KEM (X-Wing hybrid, KMS v2 wrapping) |
| **ML-DSA-65** | FIPS 204 | Validator **RT key** (Ringtail threshold) |
| **SLH-DSA** | FIPS 205 | Hash-based stateless signature |
| **X-Wing** | IETF draft | Hybrid PQ KEM (ML-KEM-768 + X25519) |
| **BLS12-381** | — | Aggregate sigs, validator-set threshold |
| **Ed25519, secp256k1** | — | Classical signing, EVM compat |

GPU-accelerated batch verification for ML-DSA, BLS, and ML-KEM.

## Threshold signing & MPC

Operational signing uses MPC — no single-party key exists in the clear:

| Scheme | Purpose | Repo |
|--------|---------|------|
| **CGGMP21** | Threshold ECDSA (BLS keygen + signing) | [`luxfi/mpc`](https://github.com/luxfi/mpc) |
| **FROST** | Threshold EdDSA (ML-DSA RT keygen + signing) | [`luxfi/mpc`](https://github.com/luxfi/mpc) |
| **Ringtail** | Threshold ML-DSA-65 framework | [`luxfi/threshold`](https://github.com/luxfi/threshold) |

Q-Chain validators require a **HybridProofOfPossession** binding both BLS and
RT (ML-DSA-65) keys atomically — the MPC daemon generates both in one DKG
session.

## Key management

Operational keys (allocator operator, gauge keeper, governance Safe co-signers)
live in [`luxfi/kms`](https://github.com/luxfi/kms) — the MPC-backed Lux KMS
service. **Never plaintext, never env files, never plaintext in databases.**
Authentication via Universal Auth (client-id/secret → short-lived bearer);
deploy CI fetches per-environment secrets through documented KMS endpoints.

Replicated state encrypted with **age** (X25519 → X-Wing PQ-upgrade path).
Storage backed by `luxfi/zapdb`; hourly snapshots + 1s incrementals to S3.

## Compliance

For deployments using regulated collateral (security tokens, regulated
yield), Liquid pairs with [`luxfi/compliance`](https://github.com/luxfi/compliance):

- Per-vault KYC level via `LiquidComplianceGate`
- IDV providers via [`hanzoai/idv`](https://github.com/hanzoai/idv) (Jumio, Onfido, Plaid)
- AML screening: OFAC SDN, EU consolidated, UK HM Treasury, PEP — plus
  optional [Jube](https://github.com/jube-suite/jube) sidecar for real-time scoring
- Travel Rule (FATF Rec. 16), CTR detection, stablecoin policies
- 20 jurisdictions, 14 frameworks (US SEC+FINRA, UK FCA, IOMFSA, EU MiCA,
  MAS, ASIC, FINMA, …)

## Container signing

Liquid container images are signed with Sigstore Cosign and attested via
in-toto:

```bash
cosign verify --certificate-identity-regexp '^https://github.com/luxfi/' \
  ghcr.io/luxfi/<service>:<tag>
```

## Vulnerability disclosure

| | |
|---|---|
| Email | `security@lux.network` |
| PGP | [`/.well-known/security.txt`](https://lux.network/.well-known/security.txt) |
| Embargo | 90 days standard |
| Bug bounty | [Immunefi](https://immunefi.com) (live with mainnet tag) |

In-scope: Liquid contracts, Transmuter, strategy adapters, position NFT,
governance gates. Out-of-scope: third-party DeFi protocols (file with the
upstream), social engineering, physical attacks.

## Ongoing

- Quarterly re-audit cadence on mainline contracts
- Per-strategy review for new adapters before classifier registration
- Public post-mortems within 14 days of any incident resolution
- Standing playbooks for contract pause, key rotation, service rollback
